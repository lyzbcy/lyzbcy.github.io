# Content & Posts 参考

## 适用场景
当任务涉及以下内容时读取本文件：
- 新建文章
- 修改已有文章
- front matter 字段调整
- 加密文章密码与提示修改
- permalink / URL 相关问题

---

## 1. front matter 字段不要想当然

### 风险
文章的 YAML front matter 决定：
- 标题
- 日期
- 分类
- 标签
- 描述
- 加密配置
- 站点生成行为

如果字段名写错、缩进错、引号错，页面可能：
- 直接构建失败
- 构建成功但页面行为异常
- URL 与预期不一致

### 正确做法
- 修改前先看同类文章的现有格式
- 尽量沿用当前仓库风格
- 改动后至少复查 front matter 顶部结构

---

## 2. 加密文章的密码字段不是“外部 JS 自动可见”

### 问题
Jekyll front matter 里的字段（例如 `password_hash`）不会自动暴露给外部 JS 文件。

### 正确做法
需要通过 Liquid 把值注入到页面里，例如：
- `{{ page.password_hash }}`
- 再在页面内联脚本中挂到 `window` 或其他运行时变量

### 接手建议
如果发现外部 JS 读不到 front matter 字段，不要去怀疑浏览器，先检查是否做了模板注入。

---

## 3. 只改密码时，优先最小修改

### 风险
只改一个 `password_hash`，却可能因为回写方式不对把整篇中文文章搞乱码。

### 最稳妥做法
- 先精确定位文章
- 只改 `password_hash` 这一项
- 不要顺手重排全文
- 如果文件已经疑似坏掉，优先从 git 历史恢复全文，再只改目标字段

---

## 4. 新文章要先观察仓库的实际 permalink 规则

### 原因
这个项目不是任意博客模板，它有自己的：
- `url`
- `baseurl`
- `permalink`
- Chirpy 文章 URL 生成方式

### 接手建议
用户要“发我链接”时，别硬猜 URL。优先检查：
- `_config.yml` 中的 `url`
- posts 的 permalink 模式
- 标题 slug 的生成结果

---

## 5. 内容型改动也要做 repo-specific 复查

### 最少复查项
- 文章标题是否正常
- 中文是否乱码
- front matter 是否闭合
- 自定义 HTML 片段是否闭合
- 图片/音频等资源地址是否仍可访问
