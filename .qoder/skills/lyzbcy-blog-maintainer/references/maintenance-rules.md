# Maintenance Rules 参考

## 适用场景
当任务涉及以下内容时读取本文件：
- 为项目补充新经验
- 设计维护规范
- 项目交接
- 想把临时经验沉淀成长期资产

---

## 1. 不再使用单文件“错题本流水账”模式

### 原因
单文件长期演化后通常会变成：
- 难搜索
- 难归类
- 难判断优先级
- 新接手的人读不动
- 改一处容易影响整体结构

### 替代方案
把经验拆成 skill + 主题参考文件：
- Skill 负责告诉接手者“什么时候用、先看什么、怎么判断”
- Reference 负责存放具体坑点和规则

这样更适合渐进式披露，也更利于后续维护。

---

## 2. 沉淀经验时优先抽象“规则”，而不是记录“流水账”

### 差的写法
- 今天我点了哪里
- 然后报了什么错
- 后来又试了哪个命令

### 更好的写法
- 什么场景会触发这个坑
- 为什么会发生
- 应该先检查什么
- 最稳的修复路径是什么

---

## 3. 新坑加入时，要放到最相关主题文件

### 路由规则
- 前端行为问题 → `frontend-and-ui.md`
- 部署 / workflow 问题 → `deploy-and-workflow.md`
- 文章 / front matter / URL 问题 → `content-and-posts.md`
- 维护方式 / 协作规范 → `maintenance-rules.md`

如果一个坑跨多个主题：
- 在主文件详细写
- 其他文件只加短链接式提示，避免重复复制

---

## 4. 交接型 skill 要强调“先判断，再展开”

### 原则
不要把所有细节都塞进 SKILL.md 本体。
SKILL.md 应该主要承担：
- 任务分类
- 阅读顺序
- 安全流程
- 决策框架

细节放进 references，按需读取。

这就是渐进式披露：
- 默认上下文轻
- 遇到对应问题再展开
- 接手成本低，维护也不容易失控

---

## 5. 每次重大修复后最好补充“如何避免再次发生”

沉淀经验时，不只写“怎么修”，还要补一句：
- 以后在什么前提下应提前警惕
- 哪一步最容易出错
- 哪种偷懒方式不要再用

这样 reference 才是可复用资产，而不是事故纪要。
